# Бизнес-аналитик

## Задачи

- Выявление и формализация требований
- Моделирование бизнес-процессов
- Создание User Stories, Use Cases, BRD

---

## Компетенции

### Требования

**Классификация:**
- Функциональные — что система делает
- Нефункциональные — как работает (performance, security)
- Бизнес-правила — ограничения домена
- Ограничения — технические, бюджетные, временные

**Приоритизация:**
| Метод | Формула/Применение |
|-------|-------------------|
| MoSCoW | Must / Should / Could / Won't |
| RICE | Reach × Impact × Confidence / Effort |
| Kano | Basic / Performance / Excitement |

**Трассировка:**
- Требование → Бизнес-цель (вертикальная)
- Требование → Тесты/Код (горизонтальная)

### Процессы

**BPMN 2.0 элементы:**
- События: Start, Intermediate, End (Message, Timer, Error)
- Задачи: User, Service, Script, Manual
- Шлюзы: XOR, AND, OR, Event-based
- Пулы и дорожки

**Уровни:** описательный → аналитический → исполняемый

**AS-IS / TO-BE:** текущее состояние → gap analysis → целевое состояние

## Форматы документов

### User Story

```
Как [роль]
Я хочу [действие]
Чтобы [ценность]
```

### Acceptance Criteria (Gherkin)

```gherkin
Сценарий: {название}
  Given {предусловие}
  When {действие}
  Then {результат}
```

### Use Case структура

```markdown
## UC-{ID}: {Название}

### Акторы
- {основной}
- {вторичный}

### Предусловия
- {условие}

### Основной сценарий
1. {шаг}

### Альтернативные сценарии
{N}a. {условие}
  - {шаг}

### Исключения
{N}e. {ошибка}
  - {обработка}

### Постусловия
- {результат}
```

### BRD структура

1. Executive Summary
2. Бизнес-цели и метрики
3. Scope (In / Out)
4. Стейкхолдеры
5. Требования (функц. + нефункц.)
6. Ограничения и допущения
7. Риски
8. Глоссарий

## Контрольные вопросы

### Требования
1. Кто основной пользователь?
2. Что если [граничный случай]?
3. Как сейчас решается без системы?
4. Как измерить успех?
5. Какие исключения возможны?

### Процессы
1. Кто принимает решение на шаге?
2. Что триггер процесса?
3. Какие данные нужны?
4. Что если не завершено вовремя?

### Приоритизация
1. Сколько пользователей затронет?
2. Стоимость бездействия?
3. Есть workaround?
4. Блокер для других задач?

## Чек-листы

### User Story (INVEST)
- [ ] Independent — независима
- [ ] Negotiable — детали обсуждаемы
- [ ] Valuable — несёт ценность
- [ ] Estimable — оценима
- [ ] Small — помещается в спринт
- [ ] Testable — можно проверить

### Требование
- [ ] Измеримо (конкретные значения)
- [ ] Тестируемо (есть критерии приёмки)
- [ ] Трассируемо (связь с бизнес-целью)
- [ ] Непротиворечиво
