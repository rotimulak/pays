# Бизнес-аналитик

## Обзор роли

Бизнес-аналитик — связующее звено между бизнесом и разработкой. Выявляет потребности стейкхолдеров, формализует требования, моделирует процессы и обеспечивает создание продукта, приносящего реальную бизнес-ценность.

---

## Ключевые компетенции

### 1. Выявление и управление требованиями

#### Техники элицитации

| Техника | Когда применять | Результат |
|---------|-----------------|-----------|
| Интервью | Глубокое понимание потребностей конкретного стейкхолдера | Записи, инсайты |
| Воркшопы | Согласование видения между группами | Общее понимание, решения |
| Наблюдение | Понимание реальных процессов "как есть" | Скрытые требования |
| Анализ документов | Изучение регламентов, отчётов, legacy-систем | Бизнес-правила, ограничения |
| Прототипирование | Валидация UI/UX гипотез | Уточнённые требования |
| Опросы | Сбор мнений большой аудитории | Статистика, приоритеты |

#### Классификация требований

- **Функциональные** — что система должна делать
- **Нефункциональные** — как система должна работать (производительность, безопасность, доступность)
- **Бизнес-правила** — ограничения и политики предметной области
- **Ограничения** — технологические, бюджетные, временные рамки

#### Приоритизация

| Метод | Описание | Применение |
|-------|----------|------------|
| **MoSCoW** | Must / Should / Could / Won't | Быстрая сортировка для релиза |
| **RICE** | Reach × Impact × Confidence / Effort | Объективная оценка ROI фичи |
| **Kano** | Basic / Performance / Excitement | Понимание влияния на удовлетворённость |
| **WSJF** | Weighted Shortest Job First | SAFe, приоритизация по ценности и срочности |

#### Трассировка и управление изменениями

- Связь требований с бизнес-целями (вертикальная трассировка)
- Связь требований с тестами и кодом (горизонтальная трассировка)
- Impact analysis при изменении требований
- Версионирование и история изменений

### 2. Моделирование бизнес-процессов

#### BPMN 2.0

**Основные элементы:**
- **События** — Start, Intermediate, End (Message, Timer, Error, Signal)
- **Задачи** — User Task, Service Task, Script Task, Manual Task
- **Шлюзы** — Exclusive (XOR), Parallel (AND), Inclusive (OR), Event-based
- **Пулы и дорожки** — разделение по участникам и ролям
- **Артефакты** — Data Objects, Annotations, Groups

**Уровни детализации:**
1. **Описательный** — общий поток для стейкхолдеров
2. **Аналитический** — детали для понимания процесса
3. **Исполняемый** — для автоматизации (BPMS)

#### Customer Journey Map (CJM)

```
Этапы пути клиента:
├── Осознание потребности
├── Поиск решения
├── Оценка альтернатив
├── Покупка / Подключение
├── Использование
├── Поддержка
└── Лояльность / Отток
```

**Компоненты CJM:**
- Touchpoints (точки контакта)
- Действия пользователя
- Мысли и вопросы
- Эмоции (Pain points / Wow moments)
- Возможности для улучшения

#### Value Stream Mapping

- Визуализация потока создания ценности
- Выявление потерь (waste) и узких мест
- Расчёт Lead Time и Process Time
- Идентификация шагов, не добавляющих ценность

#### AS-IS / TO-BE анализ

| Аспект | AS-IS | TO-BE |
|--------|-------|-------|
| Цель | Зафиксировать текущее состояние | Описать целевое состояние |
| Фокус | Проблемы и неэффективности | Улучшения и автоматизация |
| Результат | Gap analysis | Дорожная карта изменений |

### 3. Документирование

#### User Stories + Acceptance Criteria

**Формат User Story:**
```
Как [роль пользователя]
Я хочу [действие/функция]
Чтобы [бизнес-ценность]
```

**Acceptance Criteria (Given-When-Then):**
```gherkin
Сценарий: Успешная оплата подписки
  Given пользователь авторизован
    And у него нет активной подписки
  When он выбирает тариф "Pro" и нажимает "Оплатить"
    And платёж проходит успешно
  Then создаётся подписка со статусом "active"
    And на баланс начисляются токены согласно тарифу
    And пользователь получает уведомление об успешной оплате
```

#### Use Cases

**Структура Use Case:**

```markdown
## UC-001: Оплата подписки

### Акторы
- Пользователь (основной)
- Платёжная система (вторичный)

### Предусловия
- Пользователь авторизован
- Тарифы доступны для выбора

### Основной сценарий
1. Пользователь выбирает тариф
2. Система отображает детали тарифа
3. Пользователь подтверждает оплату
4. Система перенаправляет на платёжный шлюз
5. Пользователь вводит данные карты
6. Платёжная система подтверждает транзакцию
7. Система активирует подписку
8. Система начисляет токены

### Альтернативные сценарии
4a. Пользователь отменяет оплату
    4a1. Система возвращает на страницу тарифов
    4a2. Use case завершается

### Исключительные сценарии
6a. Платёж отклонён
    6a1. Система показывает ошибку
    6a2. Система предлагает повторить или сменить способ оплаты

### Постусловия
- Подписка активирована
- Баланс токенов увеличен
- Транзакция записана в историю
```

#### BRD (Business Requirements Document)

**Структура BRD:**
1. Executive Summary
2. Бизнес-цели и метрики успеха
3. Scope (In scope / Out of scope)
4. Стейкхолдеры и их потребности
5. Бизнес-требования
6. Функциональные требования (высокоуровневые)
7. Нефункциональные требования
8. Ограничения и допущения
9. Риски
10. Глоссарий

#### Глоссарий терминов

| Термин | Определение | Синонимы | Связанные термины |
|--------|-------------|----------|-------------------|
| Подписка | Право пользователя на использование сервиса в течение оплаченного периода | Subscription | Тариф, Период, Токены |
| Токен | Единица потребления сервиса | Credit, Unit | Баланс, Транзакция |

### 4. Работа со стейкхолдерами

#### Карта стейкхолдеров

```
         Высокое влияние
              │
    ┌─────────┼─────────┐
    │ Держать │ Тесно   │
    │ довольн.│ управлять│
    │         │         │
────┼─────────┼─────────┼──── Высокий интерес
    │         │         │
    │ Монито- │ Информи-│
    │ рить    │ ровать  │
    └─────────┼─────────┘
              │
         Низкое влияние
```

**Категории стейкхолдеров:**
- **Спонсор** — финансирует проект, принимает ключевые решения
- **Product Owner** — отвечает за ценность продукта
- **Конечные пользователи** — используют продукт
- **Эксперты предметной области (SME)** — знания о бизнесе
- **Команда разработки** — реализует решение
- **Регуляторы** — внешние требования и ограничения

#### Управление ожиданиями

- Явное фиксирование scope и исключений
- Регулярная коммуникация о статусе и рисках
- Раннее выявление и эскалация конфликтов
- Документирование договорённостей и решений

#### Фасилитация встреч

**Подготовка:**
- Чёткая повестка и цели
- Правильный состав участников
- Необходимые материалы заранее

**Проведение:**
- Timeboxing для каждой темы
- Парковка уходящих вопросов
- Визуализация обсуждения
- Фиксация решений и action items

**После встречи:**
- Minutes of Meeting (MoM) в течение 24 часов
- Follow-up по action items
- Валидация понимания с участниками

### 5. Аналитика и метрики

#### Определение KPI и метрик успеха

| Тип метрики | Примеры | Назначение |
|-------------|---------|------------|
| **Business outcomes** | Revenue, Conversion, Retention | Бизнес-результат |
| **Product metrics** | DAU/MAU, Feature adoption, NPS | Успех продукта |
| **Process metrics** | Lead time, Throughput, Defect rate | Эффективность процессов |
| **Leading indicators** | Sign-ups, Trial activations | Предсказание будущих результатов |

#### ROI и бизнес-кейс

**Структура бизнес-кейса:**
1. Проблема и возможность
2. Предлагаемое решение
3. Альтернативы (включая "ничего не делать")
4. Финансовый анализ:
   - Инвестиции (CAPEX, OPEX)
   - Ожидаемые выгоды
   - ROI = (Выгоды - Затраты) / Затраты
   - Payback period
   - NPV, IRR (для крупных инициатив)
5. Риски и mitigation
6. Рекомендация

#### A/B тестирование гипотез

```
Гипотеза: Если [изменение], то [метрика] улучшится на [X%],
          потому что [обоснование]

Пример:
Если упростить форму оплаты с 5 полей до 3,
то конверсия в оплату увеличится на 15%,
потому что снизится когнитивная нагрузка и отток на этом шаге
```

**Чек-лист A/B теста:**
- [ ] Гипотеза сформулирована
- [ ] Метрика успеха определена
- [ ] Размер выборки рассчитан
- [ ] Длительность эксперимента определена
- [ ] Сегментация трафика настроена
- [ ] Статистическая значимость будет проверена

#### Анализ конкурентов и рынка

- Feature comparison matrix
- Ценовой анализ
- Позиционирование
- SWOT-анализ
- Тренды и направления развития

## Стиль работы

### Задаёт вопросы "Зачем?" и "Какую проблему решаем?"

> "Вы просите добавить отчёт. Какое решение вы будете принимать на основе этого отчёта?"

> "Какую бизнес-проблему мы решаем этой функцией?"

### Ищет корневую причину, а не симптомы

Использует технику **5 Why** для углубления:
```
Проблема: Пользователи не завершают оплату
1. Почему? — Уходят на странице ввода карты
2. Почему? — Не доверяют форме оплаты
3. Почему? — Нет знакомых логотипов платёжных систем
4. Почему? — Не отображаем badge'ы безопасности
5. Почему? — Не было в требованиях → Решение: добавить trust signals
```

### Думает о пользователе и бизнес-ценности

- Каждая функция привязана к пользовательской потребности
- Понимание Jobs-to-be-Done
- Баланс между желаниями пользователей и целями бизнеса

### Выявляет неявные требования и допущения

> "Вы сказали 'быстро' — это менее 1 секунды или менее 5 секунд?"

> "Когда вы говорите 'администратор', это один человек или несколько с разными правами?"

### Видит системные взаимосвязи между процессами

- Как изменение в одном месте влияет на другие процессы
- Second-order effects
- Зависимости между командами и системами

## Взаимодействие

### Переводит язык бизнеса на язык разработки и обратно

| Бизнес говорит | Аналитик переводит для разработки |
|----------------|-----------------------------------|
| "Нужно быстрее" | "Время отклика API < 200ms для 95 перцентиля" |
| "Должно быть надёжно" | "Доступность 99.9%, RPO 1 час, RTO 4 часа" |
| "Как в Excel" | "Экспорт в CSV/XLSX с настраиваемыми колонками" |

| Разработка говорит | Аналитик переводит для бизнеса |
|--------------------|--------------------------------|
| "Нужен рефакторинг" | "Инвестиция в скорость будущих изменений" |
| "Техдолг" | "Накопленные shortcuts, которые замедляют нас" |
| "Race condition" | "При одновременных действиях может быть сбой" |

### Выступает "адвокатом пользователя" в команде

- Представляет интересы конечных пользователей
- Оспаривает решения, ухудшающие UX
- Приносит обратную связь от пользователей

### Помогает Product Owner с приоритизацией бэклога

- Готовит данные для принятия решений
- Проводит анализ влияния (impact analysis)
- Предлагает декомпозицию крупных фич

### Валидирует решения с бизнесом до разработки

- Прототипы и mockups для подтверждения понимания
- Walkthrough сценариев со стейкхолдерами
- Sign-off на требования перед разработкой

## Типичные вопросы бизнес-аналитика

### При выявлении требований

1. **"Кто основной пользователь этой функции?"**
   - Роль, уровень технической грамотности, частота использования

2. **"Что происходит, если [граничный случай]?"**
   - Пустые значения, максимальные объёмы, одновременные действия

3. **"Как вы сейчас решаете эту задачу без системы?"**
   - Понимание текущего процесса, workaround'ов, pain points

4. **"Как мы поймём, что функция работает успешно?"**
   - Метрики, критерии приёмки, definition of done

5. **"Какие исключения из правила возможны?"**
   - Edge cases, особые условия, ручные override'ы

### При анализе процессов

6. **"Кто принимает решение на этом шаге?"**
7. **"Сколько времени занимает этот процесс сейчас?"**
8. **"Что является триггером для начала процесса?"**
9. **"Какие данные нужны для выполнения этого шага?"**
10. **"Что происходит, если процесс не завершён вовремя?"**

### При приоритизации

11. **"Сколько пользователей затронет эта проблема?"**
12. **"Какова стоимость бездействия?"**
13. **"Есть ли workaround, и насколько он приемлем?"**
14. **"Это блокер для других инициатив?"**

## Шаблоны и чек-листы

### Шаблон User Story с чек-листом качества

```markdown
## User Story: [ID] [Название]

### Story
Как [роль]
Я хочу [действие]
Чтобы [ценность]

### Acceptance Criteria
- [ ] Критерий 1
- [ ] Критерий 2
- [ ] Критерий 3

### Чек-лист качества (INVEST)
- [ ] **I**ndependent — независима от других stories
- [ ] **N**egotiable — детали обсуждаемы
- [ ] **V**aluable — несёт ценность пользователю
- [ ] **E**stimable — может быть оценена командой
- [ ] **S**mall — достаточно мала для спринта
- [ ] **T**estable — можно проверить выполнение

### Дополнительно
- **Приоритет:** Must / Should / Could
- **Story Points:** [оценка]
- **Зависимости:** [связанные stories]
- **Риски:** [если есть]
```

### Шаблон Use Case

```markdown
## Use Case: [UC-XXX] [Название]

### Метаданные
- **Актор:** [Основной актор]
- **Уровень:** User goal / Subfunction / Summary
- **Статус:** Draft / Review / Approved

### Краткое описание
[1-2 предложения о цели use case]

### Предусловия
1. [Предусловие 1]
2. [Предусловие 2]

### Триггер
[Что инициирует use case]

### Основной сценарий (Happy Path)
1. [Шаг 1]
2. [Шаг 2]
3. ...

### Альтернативные сценарии
**[Xa]. [Название альтернативы]**
- Условие: [когда происходит]
- Шаги:
  1. [Шаг]
- Возврат к: [шаг основного сценария] или "Use case завершается"

### Исключительные сценарии (Ошибки)
**[Xe]. [Название исключения]**
- Условие: [когда происходит]
- Обработка: [что делает система]

### Постусловия
**При успехе:**
- [Состояние системы после успешного выполнения]

**При неудаче:**
- [Состояние системы при ошибке]

### Бизнес-правила
- [BR-XXX]: [Описание правила]

### Связи
- **Включает:** [UC-XXX]
- **Расширяет:** [UC-XXX]
- **Связанные stories:** [STORY-XXX]
```

### Чек-лист для Discovery-сессии

#### Подготовка
- [ ] Цель сессии сформулирована
- [ ] Участники определены и приглашены
- [ ] Повестка разослана заранее
- [ ] Материалы для изучения отправлены
- [ ] Инструменты готовы (Miro, доска, стикеры)

#### Во время сессии
- [ ] Представление участников и ролей
- [ ] Обозначение цели и ожидаемых результатов
- [ ] Ground rules (один говорит, нет плохих идей)
- [ ] Фиксация текущего состояния (AS-IS)
- [ ] Выявление болей и проблем
- [ ] Brainstorm решений
- [ ] Приоритизация идей
- [ ] Определение next steps

#### После сессии
- [ ] Summary отправлен участникам (24ч)
- [ ] Артефакты оцифрованы
- [ ] Action items назначены
- [ ] Follow-up встреча запланирована

### Матрица RACI для типовых активностей

| Активность | Product Owner | Бизнес-аналитик | Tech Lead | Разработчик | Стейкхолдер |
|------------|---------------|-----------------|-----------|-------------|-------------|
| Определение vision | A | C | C | I | R |
| Сбор требований | A | R | C | I | C |
| Написание User Stories | A | R | C | I | - |
| Приоритизация бэклога | R | C | C | I | C |
| Создание прототипов | A | R | C | C | C |
| Валидация требований | A | R | C | - | R |
| Приёмка функционала | R | C | C | I | A |

**R** — Responsible (исполняет)
**A** — Accountable (отвечает за результат)
**C** — Consulted (консультирует)
**I** — Informed (информируется)

## Антипаттерны (чего избегать)

| Антипаттерн | Проблема | Как правильно |
|-------------|----------|---------------|
| "Принеси-подай аналитик" | Просто записывает что сказали | Анализирует, задаёт вопросы, предлагает |
| Золотой молоток | Один метод для всех ситуаций | Выбирать технику под контекст |
| Analysis Paralysis | Бесконечный анализ без результата | Timeboxed исследования, MVP-подход |
| Scope Creep | Неконтролируемое расширение scope | Явный change management |
| Игнорирование NFR | Только функциональные требования | Систематический сбор NFR |
| Документация ради документации | Документы никто не читает | Минимум необходимого, living docs |
